Provides commands for managing database instances related to the project.

### db:setup

This command automates the creation and setup of a local PostgreSQL database using Docker. It creates a container, waits for it to be ready, and then executes the DDL (`schema.sql`) and DML (`data.sql`) scripts to build and populate the database.

**Prerequisites**
- Docker must be installed and running.
- The `psycopg2-binary` package must be installed (`pip install psycopg2-binary`).
- The `schema.sql` and `data.sql` files must exist in the specified location (usually generated by the `export` command).

**Default Configuration**

When not specified via environment variables, the following defaults are used:
- **Host**: localhost
- **Port**: 15432
- **Database**: oevk
- **User**: oevk
- **Password**: oevk

You can override these using environment variables:
```bash
export POSTGRES_HOST=localhost
export POSTGRES_PORT=15432
export POSTGRES_DB=oevk
export POSTGRES_USER=oevk
export POSTGRES_PASSWORD=oevk
```

**Common Commands**

1. **Setup the database from default script locations**:
   ```bash
   python src/cli.py db setup
   ```
   This creates a Docker container with database name, user, and password all set to `oevk`.

2. **Setup the database from custom script locations**:
   ```bash
   python src/cli.py db setup --ddl-script /path/to/my_schema.sql --dml-script /path/to/my_data.sql
   ```

3. **Force recreation of the Docker container**:
   If the `oevk` container already exists, this command will stop and remove it before creating a new one.
   ```bash
   python src/cli.py db setup --force-recreate
   ```

4. **Connect to the database**:
   ```bash
   psql -h localhost -p 15432 -U oevk -d oevk
   # Password: oevk
   ```

**CLI Parameters**

- `--ddl-script`: Path to the DDL script for table creation (default: `exports/schema.sql`).
- `--dml-script`: Path to the DML script for data insertion (default: `exports/data.sql`).
- `--force-recreate`: If specified, any existing Docker container with the name `oevk` will be removed and a new one will be created.

**Idempotent Data Loading**

The database setup is **idempotent** - safe to run multiple times without errors:

- All `INSERT` statements use `ON CONFLICT DO NOTHING`
- Running `python src/cli.py db setup` multiple times will not cause duplicate key errors
- New data is added, existing data is skipped
- No need for `--force-recreate` unless you want to start fresh

**Example:**
```bash
# First run - loads all data
python src/cli.py db setup
# ✓ Database setup completed successfully.

# Second run - skips existing data, adds any new data
python src/cli.py db setup
# ✓ Database setup completed successfully.

# No errors, no duplicates!
```

**Troubleshooting**

- **"Docker is not running"**: Ensure the Docker daemon is active.
- **"psycopg2-binary is not installed"**: Run `pip install psycopg2-binary` to install the required Python driver for PostgreSQL.
- **"Could not connect to the PostgreSQL database"**: The Docker container may still be initializing. The script waits and retries, but if it fails, wait a few more seconds and try again. You can also check the container logs with `docker logs oevk`.
- **Script not found**: Ensure you have run the `export` command first to generate the `.sql` files.
- **"Duplicate key violation"**: This should no longer occur as of the idempotent update. If you still see this, ensure you're using the latest version with `ON CONFLICT DO NOTHING` in the SQL files.

**Advanced Features**

The PostgreSQL schema includes performance optimizations for text searching:

- **Trigram Indexes**: GIN indexes on `FullAddress` columns enable efficient substring searches
  ```sql
  -- Search for addresses containing "Bar" (case-insensitive)
  SELECT * FROM Address WHERE FullAddress ILIKE '%Bar%';
  
  -- Search for street types like "utca"
  SELECT * FROM Address WHERE FullAddress ILIKE '%utca%';
  ```

- **pg_trgm Extension**: Automatically enabled for fuzzy text matching
- **Canonical Data Only**: Address table contains cleansed/deduplicated data (3.3M records)
- **OriginalAddressCount**: Track deduplication ratio per canonical address
  ```sql
  -- Find addresses with high deduplication ratios
  SELECT FullAddress, OriginalAddressCount 
  FROM Address 
  WHERE OriginalAddressCount > 5 
  ORDER BY OriginalAddressCount DESC;
  ```

**PostgreSQL Schema Structure**

The export contains **13 tables** with canonical (cleansed) data:

**Entity Tables (10):**
- County, Settlement, NationalIndividualElectoralDistrict, SettlementIndividualElectoralDistrict
- PostalCode, PostalCode_Settlement, PollingStation
- PublicSpaceName, PublicSpaceType, SettlementPublicSpaces

**Address Table (1):**
- Address (~3.3M canonical/cleansed addresses with all foreign keys and OriginalAddressCount)

**Reference Tables (2):**
- AddressPollingStations (Address → PollingStation)
- AddressPIRCodes (Address → PIR codes)

**Not Exported (internal transformation tables):**
- AddressMapping, DeduplicationReport, Address_new (these are excluded from PostgreSQL export)
